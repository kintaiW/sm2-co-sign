## Repository Overview

SM2 协同签名 (SM2 Co-Sign) is a cooperative signing system based on the Chinese national cryptographic standard SM2 elliptic curve algorithm. The system splits private keys into two shares (D1 on the client, D2 on the server) so the full private key never exists in one place.

The repo is a **monorepo with two git submodules**:
- `backend/` → `https://github.com/kintaiW/sm2-co-sign-server` (Go)
- `client/` → `https://github.com/kintaiW/sm2-co-sign-client` (Rust)

The `frontend/` directory has been removed. The system now uses CLI tools for management.

## Architecture

### Key Split Protocol

The complete private key `d = d1 * d2 - 1` is never stored anywhere.

- **Client** holds: `d1` (random scalar), `userId`, `Pa` (cooperative public key)
- **Server** holds: `d2`, `d2Inv = d2⁻¹ mod n`, `Pa`
- **Public key**: `Pa = d2Inv * P1 + (n-1) * G` where `P1 = d1 * G`

### Backend (Go)

Entry point: `backend/cmd/server/main.go`
Config: `backend/config.yaml` (loaded via Viper; path can be passed as CLI arg)

Layer structure under `backend/internal/`:
- `crypto/` — SM2 cooperative algorithm (`sm2_coop.go`). Uses `github.com/emmansun/gmsm`.
- `handler/` — Fiber HTTP handlers (`user.go`, `cosign.go`, `admin.go`)
- `service/` — Business logic (`user_service.go`, `cosign_service.go`, `admin_init.go`)
- `repository/` — SQLite data access (`user_repo.go`, `key_repo.go`, `audit_log_repo.go`, `session_repo.go`)
- `middleware/` — Auth middleware (Bearer token extraction, `GetUserID` context helper)
- `model/` — Domain models
- `config/` — Config struct and loader

Routes split into two groups:
- `/api/*` — end-user operations (register, login, key/init, sign, decrypt). Most require Bearer token auth.
- `/mapi/*` — admin operations (users, keys, logs, health, stats). No auth middleware in current code.

Database is SQLite at `./data/cosign.db`; schema initialized from `scripts/schema.sql` on startup.

### Client (Rust)

Workspace root: `client/Cargo.toml`

| Crate | Purpose |
|---|---|
| `sm2_co_sign_core` | Core protocol: `CoSignProtocol` struct implements D1 generation, P1 calculation, signing steps (Q1, final s), decryption steps (T1, shared key). Uses `libsm` for raw EC ops and `gm-sdk-rs` for SM2 sign/verify and SM3. |
| `sm2_co_sign_cli` | CLI wrapper around core (via `clap`) |
| `sm2_co_sign_ffi` | C FFI bindings; header generated by `cbindgen` into `sm2_co_sign_ffi.h` |

Key files: `sm2_co_sign_core/src/protocol.rs` (protocol logic), `sm2_co_sign_core/src/client.rs` (HTTP client to backend), `sm2_co_sign_core/src/types.rs` (shared types).

Dependency note: two SM2 libraries are used intentionally — `libsm` for raw EC point operations needed by the cooperative protocol, `gm-sdk-rs` (custom fork) for standard SM2 sign/verify and SM3 where the API is simpler.

### Frontend (React + TypeScript)

Vite + React 18 + Ant Design 5 + Zustand (state) + Axios + ECharts

`frontend/src/` structure:
- `pages/` — `dashboard/`, `users/`, `keys/`, `logs/`, `sign/`, `login/`
- `stores/` — Zustand stores
- `api/` — Axios-based API clients
- `router/` — React Router v6 config
- `layouts/`, `components/`, `types/`, `utils/`

The frontend is a **management/admin UI** (not the cryptographic client). It talks to the `/mapi/*` endpoints for admin tasks and `/api/*` for signing demos.

## Submodule Workflow

Changes to `backend/` or `client/` must be committed and pushed in their own repos first, then the parent repo updated:

```bash
cd backend   # or client
git add . && git commit -m "..." && git push
cd ..
git add backend   # or client
git commit -m "Update backend submodule"
```
